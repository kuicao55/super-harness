# Harness Initializer

Packages the current session state into a Handoff Document, then triggers `/clear` to start a fresh context. Used to preserve state across sessions, prevent context pollution, and enable clean resume.

**Announce at start:** "I'm using the harness-initializer skill to create a handoff document and reset the session."

## Trigger Conditions

This skill is called when ANY of these occur:

1. **Milestone completion** — current milestone's last task passed Code Quality Review
2. **Context threshold** — Orchestra has executed 5+ consecutive Tasks without reset
3. **User manual call** — user invokes `/harness:initialize`
4. **Resume handoff** — `docs/harness/handoffs/` contains a handoff document to load

## Step 1: Gather Session State

Before writing the handoff document, gather:

1. **Read `status/claude-progress.json`** — current milestone, passed tasks, remaining tasks
2. **Read recent activity log** — `logs/activity-*.jsonl` for current session's entries
3. **Check active git worktree** — `git worktree list` to note any worktree branches in use
4. **Read current plan file** — `docs/harness/plans/*.md` for in-progress milestone plan
5. **Note recent decisions** — any significant technical decisions made during this session

## Step 2: Write Handoff Document

Save to: `docs/harness/handoffs/YYYY-MM-DD-HH-MM.md`

```markdown
# Handoff Document — <YYYY-MM-DD HH:MM>

## Current Milestone

- **Milestone:** <milestone-id> — <title>
- **Status:** <in-progress | completed>
- **Plan file:** <path/to/plan.md>

## Completed Tasks

| Task | Verdict | Key Files | Notes |
| ---- | ------- | --------- | ----- |
| Task N | PASS | file.py, test_file.py | brief summary |
| ... | ... | ... | ... |

## Pending Tasks

| Task | Status | Blocked By |
| ---- | ------ | ---------- |
| Task N+1 | pending | — |
| ... | ... | ... |

## Failed / Blocked Tasks (This Session)

- **Task N:** FAIL (Code Quality Review) — [brief reason]
- **Task N:** BLOCKED — [reason and what was tried]

## Active Worktree

- Branch: `<branch-name>` at `<worktree-path>` (if applicable)

## Significant Decisions

- [Decision 1]: [context and rationale]
- [Decision 2]: [context and rationale]

## Next Steps

1. [Immediate next action]
2. [Subsequent action]
3. [Long-term note]

## Resume Command

```
/harness:resume
```

---

*Generated by harness-initializer on <YYYY-MM-DD HH:MM>*
```

## Step 3: Confirm with User

Show the handoff document summary and ask:

> "Handoff document created. Ready to clear the session and start fresh? (yes/no)
>
> On resume, I'll load this document automatically via `/harness:resume`."

- **yes** → proceed to Step 4
- **no** → abort. Do not write the handoff document. Continue the session.

## Step 4: Reset Context

After user confirmation:

1. Announce: "Clearing session context. Your next session can resume with `/harness:resume`."
2. Execute `/clear` (via the CLI command mechanism)
3. The next session's `/harness:resume` call will find the handoff document and load it

## Key Constraints

- **Read-only:** Initializer never modifies code, never creates files outside `docs/harness/handoffs/`
- **No judgments:** Does not evaluate quality or correctness — only packages state
- **No engineering decisions:** Does not recommend approaches or priorities — records what was decided
- **User confirms:** Never auto-resets without explicit user confirmation

## Integration

This skill is referenced by:
- `harness-entry/SKILL.md` — `/harness:initialize` route
- `harness-execution/SKILL.md` — milestone completion and context threshold triggers
